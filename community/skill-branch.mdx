# SKill 分支文档

NEC 主仓库在 Gitee 上维护了一个 **`SKill` 分支**，用于沉淀与整理技能学习、知识分享、教程资料等内容。本页面作为该分支的文档入口，方便新人快速定位学习资源，也方便贡献者了解如何向分支提交资料。

## 分支定位

`SKill` 分支专门存放**非代码资产**：

- 技能教程与学习笔记
- 竞赛经验复盘与踩坑记录
- 工具链使用说明
- 项目总结与方法论沉淀
- 新人入门指引的补充资料

> **为什么单独一个分支？** 代码仓库的主分支以工程交付为主，而技能资料更新频繁、版本要求较低，单独分支可以避免污染主分支的提交历史，同时让学习资料更容易被检索和复用。

```text

│   ├── vision/

│   ├── git/
│   ├── python/
```

> 实际目录以 Gitee 分支上的最新文件为准。如果你发现某个领域资料缺失，欢迎补充。

## 如何查找资料

1. **按技术领域浏览**：进入对应目录，查找 `README.md` 或专题文档。
2. **按关键词搜索**：在 Gitee 仓库内使用搜索功能，限定 `SKill` 分支。
3. **从本页跳转**：下方会持续更新已有资料的索引卡片。

## 建议贡献内容

如果你有想要分享的技能资料，欢迎向 `SKill` 分支提交 PR。以下内容都受欢迎：

### 机械方向
- SolidWorks / Fusion 360 建模教程
- 3D 打印参数与后处理经验
- 加工工艺（CNC、线切割、钣金）入门
- 常用标准件选型指南

### 嵌入式方向
- STM32 / ESP32 开发环境配置
- 电机驱动（CAN、PWM、FOC）笔记
- 传感器使用教程（IMU、摄像头、激光雷达）
- RTOS 与实时控制心得

### 算法方向
- 视觉识别（OpenCV、YOLO、传统算法）
- 路径规划与运动控制
- SLAM 入门资料
- 数据集标注与训练经验

### 软件与工具链方向
- Git 工作流与提交规范
- Python 工程化实践
- ROS2 节点开发与调试
- 飞书、Gitee、Markdown 使用技巧

### 竞赛与项目方向
- 赛季复盘（做了什么、为什么失败、下次怎么改）
- 规则解读与策略分析
- 备赛时间线与任务拆分模板
- 预算申报与采购流程经验

## 贡献流程

<Steps>
  <Step title="Fork 或拉取仓库">
    在 Gitee 上 Fork NEC 主仓库，或本地克隆后切换到 `SKill` 分支。

    ```bash
    git clone https://gitee.com/darrenpig/new_energy_coder_club.git
    cd new_energy_coder_club
    git checkout SKill
    git pull origin SKill
    ```
  </Step>

  <Step title="在对应目录添加资料">
    按技术领域放到对应目录。如果是新领域，可以新建目录并在根目录 `README.md` 中说明。

    ```text
    SKill/embedded/stm32/
    └── README.md
    ```
  </Step>

  <Step title="遵循格式规范">
    - 使用 Markdown 格式
    - 文件名使用英文或拼音，避免空格
    - 图片统一放到同目录 `images/` 下，使用相对路径引用
    - 在文档开头写明作者、日期、适用版本
  </Step>

  <Step title="提交 PR">
    向 `SKill` 分支发起 Pull Request，标题建议：

    ```text
    docs(skill): 添加 STM32 开发环境配置指南
    ```

    PR 描述中说明资料内容、目标读者和参考资料。
  </Step>
</Steps>

## 常见问题

<AccordionGroup>
  <Accordion title="SKill 分支和 Wiki 有什么区别？">
    SKill 分支是仓库内的 Markdown 资料集，适合版本化、结构化的教程和笔记；Wiki（Mintlify 文档站）面向对外展示和社区导航。两者可以互补：SKill 沉淀原始资料，Wiki 做精选汇总。
  </Accordion>

  <Accordion title="我可以直接修改别人的资料吗？">
    可以，但建议通过 PR 提交修改，并在描述中说明改动原因。如果是大幅改写，最好先在群里或 Issue 中与原作者沟通。
  </Accordion>

  <Accordion title="资料需要多高的完成度才能提交？">
    不追求一次完美。即使是半成品笔记，只要标注了"草稿"状态和已知缺口，对新人也可能有价值。社区会一起迭代完善。
  </Accordion>
</AccordionGroup>
